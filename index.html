About this demo

A deep dive into llm-d's Cache Manager and Tiered KV Cache Offloading

This demo focuses on two specific features of the llm-d project: the KV-Cache Indexer for cache-aware routing, and tiered KV cache offloading across GPU, CPU, and persistent storage.

When you run large AI models across multiple servers, a lot of repeated computation happens behind the scenes. These features eliminate that waste by caching intermediate results and routing requests to the server most likely to already have relevant cached data.

No AI/ML expertise required to follow along.

In plain terms

Why does this matter?

When you send a message to an AI, the model has to read and process your entire prompt before it can start responding. This step is called prefill.

Most enterprise apps use a system prompt — a long set of instructions that's the same for every user. Think: "You are a helpful customer service agent for Acme Corp..."

Imagine your F1 team has two cars entering qualifying. Each needs the same aerodynamic setup dialed in. Without a strategist coordinating, each engineer starts from zero — measuring, adjusting, testing — doing identical work twice. With a shared setup sheet, one engineer dials it in and the other just copies it.

Without cache management, the load balancer has no idea which server already processed that system prompt — so it sends each request to a random pod, and every pod re-does the same expensive work.

Bottom line: You're paying 3x the GPU compute for no reason. At scale, this is the difference between 10 GPUs and 30.

In plain terms

How does the system recognize repeated text?

Every prompt gets broken into small chunks of tokens (roughly words). Each chunk gets a unique fingerprint — a short code that identifies it.

If two requests start with the same words, their fingerprints will match exactly. The system sees the match and says: "I already computed this part — I can skip it."

Think of it like tire telemetry data in F1. Each set of conditions (track temp, tire compound, fuel load) gets a unique signature. If a car encounters the same conditions again, the team already knows the optimal setup — no need to run practice laps again.

The "extra" field is a safety mechanism. It adds a namespace (like a tenant ID or model adapter name) to the hash, so two different customers with the same prompt don't accidentally share cached data.

Bottom line: Fingerprinting makes prefix reuse fast and safe. No data is transferred to check — just a quick hash comparison.

In plain terms

The central directory

The indexer is like a library catalog. It doesn't hold the actual data — it just knows which server has which cached blocks, and whether they're in fast memory or slower storage.

Like a race strategist with a telemetry dashboard — they know which car has warm tires, which has fresh fuel, and which pit crew is ready. When the next pitstop call comes, they route the car to the crew that's already prepped.

Events stream in real-time. The index is best-effort, not a guarantee — but close enough for smart routing decisions.

Bottom line: One central service knows what's cached where, so routing can be intelligent instead of random.

In plain terms

Same 3 requests — totally different outcome

This is the same scenario from slide 2: three users send requests with the same system prompt. But now the router is cache-aware.

Request 1 still has to do full prefill — nothing is cached yet. But the router sends it to one specific pod, which caches the result.

Requests 2 and 3 arrive and the router checks the index: "Pod 0 already has this prefix cached in GPU." It routes both there — and they skip prefill entirely.

Back to the F1 race: the first car still does the formation lap. But now cars 2 and 3 are sent to the same pit crew that already has warm tires and fuel ready — they skip the formation lap and go straight to racing.

Watch the TTFT counters on the left — the first request takes ~850ms, but the cached requests complete in ~52ms. That's a 16x improvement just from routing smarter.

Bottom line: Same 3 requests. Slide 2 wasted 2,550ms of GPU compute. Here, total TTFT is 954ms — a 63% reduction, with zero hardware changes.

In plain terms

A three-level cache

GPU memory is blazing fast but tiny and expensive (~80 GB per GPU). When it fills up, older cached data doesn't just vanish — it moves down to a cheaper, bigger tier.

Tier 1 — GPU memory: Fastest. This is where active inference happens. Like the top of your desk — everything is within arm's reach.

Tier 2 — CPU memory: Still fast. Transfers happen asynchronously in the background, so inference isn't blocked. Like a filing cabinet beside your desk.

Tier 3 — Shared storage: Slowest, but massive, cheap, and persistent. Data survives pod restarts. Any pod in the cluster can access it. Like a warehouse down the street.

Think of it like F1 tire strategy: hot tires on the car (GPU) are fastest. Pre-warmed spares in the pit lane (CPU) are ready in seconds. Cold spares in the garage (storage) take longer to prep — but still 16.8x faster than manufacturing new tires mid-race.
Bottom line: Instead of throwing away cached data when GPU is full, it cascades to cheaper storage. More cache = more hits = less wasted compute.

In plain terms

Reading the numbers

TTFT (Time to First Token) is how long a user waits before the AI starts typing back. Lower is better — it's the "thinking..." delay.

Throughput is how many tokens (roughly words) the whole system processes per second across all users. Higher means you can serve more people.

The yellow bars show the baseline: vLLM with GPU + CPU caching only. The green bars show what happens when you add a shared storage tier on top.

Like sharing telemetry data across the whole racing fleet. Any car in the team can benefit from data collected by any other car. The more laps (longer prompts) the data covers, the bigger the advantage.

The gains are strongest with long system prompts (50K-70K tokens) because there's more reusable prefix to cache. At 70K tokens, the storage tier saves ~13 seconds of user wait time per request.

Bottom line: 20-29% more throughput means you can serve the same traffic with fewer GPUs — or handle 29% more users with the same hardware.
